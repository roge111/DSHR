# Этап 1
Для начала необходимо создать резервную копию на основном узле. Для этого имеется скрипт `backup_script.sh`.

Он представлен ниже:

```
#!/bin/bash

# Конфигурация
PGPORT=9776
PGUSER=postgres4
BACKUP_DIR="$HOME/backups/base"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS_LOCAL=7
REMOTE_HOST="postgres4@199"
REMOTE_BACKUP_DIR="~/backups/base"
LOG_FILE="$HOME/backup.log"

echo "[$(date)] === НАЧАЛО БЭКАПА ===" >> $LOG_FILE

# pg_basebackup с включением WAL
pg_basebackup -h localhost -p $PGPORT -U $PGUSER \
    -D $BACKUP_DIR/base_backup_$DATE \
    -Ft -z -P -X fetch 2>> $LOG_FILE

if [ $? -eq 0 ]; then
    echo "[$(date)] Бэкап создан: base_backup_$DATE" >> $LOG_FILE
    
    # Копирование на резервный узел
    ssh $REMOTE_HOST "mkdir -p $REMOTE_BACKUP_DIR" 2>> $LOG_FILE
    scp $BACKUP_DIR/base_backup_$DATE.tar.gz $REMOTE_HOST:$REMOTE_BACKUP_DIR/ 2>> $LOG_FILE
    
    # Удаление старых бэкапов на основном узле (старше 7 дней)
    find $BACKUP_DIR -name "base_backup_*.tar.gz" -mtime +$RETENTION_DAYS_LOCAL -delete
else
    echo "[$(date)] ОШИБКА создания бэкапа!" >> $LOG_FILE
fi

echo "[$(date)] === ЗАВЕРШЕНИЕ ===" >> $LOG_FILE
```

## Разъяснение строк скрипта backup_script.sh

1. **`#!/bin/bash`** — указывает, что скрипт должен выполняться в интерпретаторе Bash.
2. **`PGPORT=9776`** — задает порт, на котором работает PostgreSQL.
3. **`PGUSER=postgres4`** — имя пользователя PostgreSQL для подключения.
4. **`BACKUP_DIR="$HOME/backups/base"`** — директория на основном узле, куда будут сохраняться бэкапы.
5. **`DATE=$(date +%Y%m%d_%H%M%S)`** — генерирует строку с текущей датой и временем в формате ГГГГММДД_ЧЧММСС для уникального имени архива.
6. **`RETENTION_DAYS_LOCAL=7`** — срок хранения бэкапов на основном узле (7 дней).
7. **`REMOTE_HOST="postgres4@199"`** — адрес резервного узла (имя пользователя и хост).
8. **`REMOTE_BACKUP_DIR="~/backups/base"`** — директория на резервном узле для хранения бэкапов.
9. **`LOG_FILE="$HOME/backup.log"`** — файл журнала, куда записываются события скрипта.
10. **`echo "[$(date)] === НАЧАЛО БЭКАПА ===" >> $LOG_FILE`** — записывает в лог-файл отметку о начале операции.
11. **`pg_basebackup -h localhost -p $PGPORT -U $PGUSER \`** — команда создания полной резервной копии PostgreSQL с указанием хоста, порта и пользователя.
12. **`-D $BACKUP_DIR/base_backup_$DATE \`** — задает целевую директорию для бэкапа с уникальным именем.
13. **`-Ft -z -P -X fetch 2>> $LOG_FILE`** — параметры:
    - `-Ft` — создает архив в формате tar.
    - `-z` — сжимает архив с помощью gzip.
    - `-P` — показывает прогресс выполнения.
    - `-X fetch` — включает WAL (Write‑Ahead Log) файлы в бэкап, что необходимо для восстановления.
    - `2>> $LOG_FILE` — перенаправляет ошибки (stderr) в лог-файл.
14. **`if [ $? -eq 0 ]; then`** — проверяет код возврата предыдущей команды (pg_basebackup). Если 0 (успех), выполняет блок then.
15. **`echo "[$(date)] Бэкап создан: base_backup_$DATE" >> $LOG_FILE`** — записывает в лог сообщение об успешном создании бэкапа.
16. **`ssh $REMOTE_HOST "mkdir -p $REMOTE_BACKUP_DIR" 2>> $LOG_FILE`** — подключается по SSH к резервному узлу и создает директорию для бэкапов (если её нет).
17. **`scp $BACKUP_DIR/base_backup_$DATE.tar.gz $REMOTE_HOST:$REMOTE_BACKUP_DIR/ 2>> $LOG_FILE`** — копирует архив бэкапа на резервный узел через SCP.
18. **`find $BACKUP_DIR -name "base_backup_*.tar.gz" -mtime +$RETENTION_DAYS_LOCAL -delete`** — удаляет на основном узле архивы старше 7 дней.
19. **`else`** — если pg_basebackup завершился с ошибкой, выполняется блок else.
20. **`echo "[$(date)] ОШИБКА создания бэкапа!" >> $LOG_FILE`** — записывает в лог сообщение об ошибке.
21. **`fi`** — завершает условный оператор if.
22. **`echo "[$(date)] === ЗАВЕРШЕНИЕ ===" >> $LOG_FILE`** — записывает в лог отметку о завершении работы скрипта.

Затем настроим очистку резервного узла. Это находится в файле `cleanup_script.sh`.

Вот его скрипт:

```
#!/bin/bash

BACKUP_DIR="$HOME/backups/base"
RETENTION_DAYS=30
LOG_FILE="$HOME/cleanup.log"

echo "[$(date)] Очистка бэкапов старше ${RETENTION_DAYS} дней" >> $LOG_FILE

find $BACKUP_DIR -name "base_backup_*.tar.gz" -mtime +$RETENTION_DAYS -delete

COUNT=$(ls -1 $BACKUP_DIR/base_backup_*.tar.gz 2>/dev/null | wc -l)
SIZE=$(du -sh $BACKUP_DIR 2>/dev/null | cut -f1)

echo "[$(date)] Осталось бэкапов: $COUNT, размер: $SIZE" >> $LOG_FILE
```

## Разъяснение строк скрипта cleanup_script.sh

1. **`#!/bin/bash`** — указывает, что скрипт должен выполняться в интерпретаторе Bash.
2. **`BACKUP_DIR="$HOME/backups/base"`** — директория, где хранятся бэкапы на резервном узле.
3. **`RETENTION_DAYS=30`** — срок хранения бэкапов на резервном узле (30 дней).
4. **`LOG_FILE="$HOME/cleanup.log"`** — файл журнала для записи событий очистки.
5. **`echo "[$(date)] Очистка бэкапов старше ${RETENTION_DAYS} дней" >> $LOG_FILE`** — записывает в лог сообщение о начале очистки с указанием срока хранения.
6. **`find $BACKUP_DIR -name "base_backup_*.tar.gz" -mtime +$RETENTION_DAYS -delete`** — находит и удаляет архивы бэкапов, которые старше 30 дней.
    - `-name "base_backup_*.tar.gz"` — шаблон имени файлов.
    - `-mtime +$RETENTION_DAYS` — файлы, измененные более $RETENTION_DAYS дней назад.
    - `-delete` — удаляет найденные файлы.
7. **`COUNT=$(ls -1 $BACKUP_DIR/base_backup_*.tar.gz 2>/dev/null | wc -l)`** — подсчитывает количество оставшихся бэкапов.
    - `ls -1` — выводит по одному файлу на строку.
    - `2>/dev/null` — подавляет сообщения об ошибках (например, если файлов нет).
    - `wc -l` — считает количество строк (т.е. файлов).
8. **`SIZE=$(du -sh $BACKUP_DIR 2>/dev/null | cut -f1)`** — вычисляет общий размер директории с бэкапами.
    - `du -sh` — показывает размер директории в удобочитаемом формате (например, "1.2G").
    - `cut -f1` — извлекает первое поле (размер) из вывода du.
9. **`echo "[$(date)] Осталось бэкапов: $COUNT, размер: $SIZE" >> $LOG_FILE`** — записывает в лог итоговую статистику после очистки.

Приступим к настройкам расписания хранения резервного копирования. В основном узле используем команду `crontab -e` и затем вводим следующую строку
```0 2,14 * * * /home/postgres4/backup_script.sh```

Согласно синтаксису это означает:
- `0` — минута
- `2,14` — час (два раза: в 2 часа и в 14 часов)
- `*` — день месяца (каждый день)
- `*` — месяц (каждый месяц)
- `*` — день недели (каждый день недели)
- `/home/postgres4/backup_script.sh` — путь к скрипту

Таким образом настроено расписание резервного копирования в основном узле каждый день в 2 и 14 часов.


С помощью этих настроек и скриптов настроено автоматическое уничтожение файлов старше 7 дней на основном узле и старше 30 дней на резервном узле.

На основном узле выполняется командой `find -mtime +7 -delete`, на резервном — `find -mtime +30 -delete`

## Расчеты

Предположим, что БД в первый день пустая - 0 ГБ

Мы имеем:
 - Исходные данные в сутки = 1000 МБ
 - Измененные данные в сутки = 100 МБ
 - Частота бэкапов = 2 раза в сутки
 - Хранение на основном узле = 7 суток
 - Хранение на резервном узле = 30 суток


Каждый день добавляется суммарно 1100 МБ данных — это примерно 1.1 ГБ данных

Тогда размер базы данных вычисляется так:
```
S(d) = 1.1 * d
```
Где `d` — номер дня. Например, на 30-й день размер будет достигать 33 ГБ

Размер одной резервной копии равен размеру БД: ``` V(d) = S(d) ```

Рассчитаем теперь объем резервных копий на основном узле. Тут важно учитывать, что размер зависит от того, с какого по какой день сохранялись резервные копии. С каждым днем объем резервной копии увеличивается, так как увеличивается сама БД. Возьмем с 7-го по 14-й
```
S(7) = 7.7 ГБ
S(8) = 8.8 ГБ
S(9) = 9.9 ГБ
S(10) = 11 ГБ
S(11) = 12.1 ГБ
S(12) = 13.2 ГБ
S(13) = 14.3 ГБ


```
Общий объем при одной резервной копии в день:
```
77 ГБ
```
При условии, что копирование выполняется два раза в сутки:
```
154 ГБ
```

Но если брать, например, с 24-го по 30-е, то размер будет равным `415.8 ГБ` — это месяц спустя работы!

Ответ: 415.8 ГБ



Рассчитаем объем на резервном узле:
```
Σ V(d) = Σ (1.1 × d) = 1.1 × Σ d   для d = 1..30
```

Просуммируем d:
```
1 + 2 + 3 + ... + 30 = 465
```
Значит:
```
Σ V(d) = 1.1 × 465 = 511.5 ГБ
```

И это при одной копии в день, а у нас — две
```
2 × 511.5 = 1023 ГБ
```
Ответ: 1023 ГБ
